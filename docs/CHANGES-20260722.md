## 2026-07-22 09:30 MSVC inplace stub TLS_PROXY 模式修复（marker/function 节分离）

### 修复

**MSVC 构建的 inplace stub 在有 TLS callbacks 的样本上崩溃 0xC0000005**

现象：
- MSVC 构建的 inplace stub（无 MinGW 覆盖）在 hellomingw.exe / helloucrt.exe 上崩溃
  0xC0000005（这两个样本有 TLS callbacks，触发 TLS_PROXY 模式）
- hellocli.exe（无 TLS）正常，其他 GUI 样本（Notepad4、DontSleep）也正常
- reflective stub 不受影响（用不同的 TLS 处理方案）

根因：MSVC link.exe 不合并不同 flag 的 `$` 子节
- `.lock$tlscbm` 是 const data 段（flags 0x40000040，只读不可执行）
- `.lock$tlscb` 是 code 段（flags 0x60000020，可执行）
- 两者 flag 不同，link.exe 不会把它们合并到同一个输出节
- 导致 MSVC stub.exe 出现 4 个独立 .lock 节，marker 和 function 分到不同节
- [builder.c](file:///C:/Home/Projects/applocker/packer/inplace/builder.c) 的
  `find_stub_tls_cb_offset` 假设 `function = magic + 16`，但 magic+16 处是节末填充（全零）
- builder 把错误地址写入 TLS callbacks 数组，OS loader 调用错误地址 → 0xC0000005

修复方案：用 .asm 文件把 marker 和 function 放进同一个 `.lock$tlscb` SEGMENT
- [stub_asm_x64.asm](file:///C:/Home/Projects/applocker/packer/inplace/stub_asm_x64.asm)：
  新增 `locktlscb SEGMENT READ EXECUTE ALIAS('.lock$tlscb')`，包含 marker + stub_tls_callback
- [stub_asm_x86.asm](file:///C:/Home/Projects/applocker/packer/inplace/stub_asm_x86.asm)：
  同上，x86 版本用 `ret 12`（stdcall 3 个 DWORD 参数），符号名 `_stub_tls_callback@12`
- [stub.c](file:///C:/Home/Projects/applocker/packer/inplace/stub.c)：
  用 `#ifndef _MSC_VER` 包裹 marker 和 stub_tls_callback 的 C 定义
  （MinGW 仍用 C 定义 + stub.ld 的 KEEP() 保证顺序）
- [CMakeLists.inc](file:///C:/Home/Projects/applocker/packer/inplace/CMakeLists.inc)：
  添加 `/INCLUDE:stub_tls_callback`（x64）或 `/INCLUDE:_stub_tls_callback@12`（x86）
  强制保留 stub_tls_callback（未被 stub.c 直接引用，/OPT:REF 会移除）

验证：
- stub.bin 布局验证：magic+16 处是 stub_tls_callback 的 ret 指令
  - x64: `C3` (ret)
  - x86: `C2 0C 00` (ret 12)
- auto e2e test 全部通过：18/18（9 inplace + 9 reflective）
  - 包括之前失败的 hellomingw.exe / helloucrt.exe（TLS_PROXY 模式）
- 现在 MSVC 构建的 inplace stub 完全可用，不再需要 MinGW 覆盖

影响：packer 的 MSVC 迁移工作完成，inplace stub 可完全用 MSVC 构建
